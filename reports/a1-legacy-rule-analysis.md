## Solar Invoice Route Analysis (routes_admin.py:627-749)

The current `solar_invoice()` function produces a minimum of 11.00 EGP:
```python
admin_fee = round(amount * 0.02, 2)  # 2% of consumption amount
service_fee = 11                       # Fixed service fee
total = round(amount + admin_fee + service_fee, 2)
```

At zero consumption: `total = 0 + 0 + 11 = 11.00 EGP`.

The download function (line 700+) reads fees from invoice notes, defaulting to `service_fee = 11`. Historical invoices have empty notes, suggesting they were generated by an OLDER version of this code.

## Evidence of Code Versioning

1. **All 2,797 invoices** have `notes=''` (empty) — The current code would populate notes with reading data (`قراءة 1.8.0: ...`). Empty notes prove these invoices were generated by an earlier code version.

2. **The 36.10 minimum appears in 2021-01 through 2022-02 only** — Clean transition at Feb 2022 suggests a deliberate code/tariff change on a specific date.

3. **The same 16 customers that had 36.10 now get 9.10** — After Feb 2022, their minimum months all drop to 9.10. If this were a customer-level tariff assignment issue, some would have stayed at 36.10.

4. **Customers with highest consumption (52051440, 52051442) never showed 9.10** — Because they never had a zero-consumption month after Feb 2022. Their post-2022 minimums (53.36 and 154.71) are consumption-based, not minimum-based.

## Legacy Rule Conclusion

The 36.10 EGP minimum was a **hardcoded legacy business rule** in the solar billing code. It was:

1. **Not derived from tariff tiers** — The solar consumption tiers are identical to electricity tiers and cannot produce 36.10 from zero consumption.
2. **Not derived from known fee templates** — No template (residential, commercial, construction, lighting, water pump, service) produces 36.10.
3. **Not in the tariff_charge table** — The tariff_charge table is empty. No STATIC or ZERO charges exist.
4. **A deliberate business rule** — The exact same 36.10 value across all 16 customers for 14 months, with a clean transition to 9.10 in Feb 2022.

The service fee was hardcoded as `service_fee = 36.10` in the legacy code, then changed to `service_fee = 9.10` around Feb 2022, and finally changed to `service_fee = 11` in the current Meter Verse code.

## SBill Schema Reference

The SBill Palm Hills schema shows a more complex tariff structure:
```
tariff (id, code, name, description, effective_date, expiry_date, status)
tariff_charges (id, tariff_id, charge_group, charge_type, flat_amount, flat_rate, per_unit_rate, upper_limit, recurring_mode)
tariff_charge_details (id, charge_id, from_value, to_value, rate)
```

The charge_group field (0=Consumption, 1=Service Fee, 2=Tax) matches the Meter Verse schema. However, none of the Meter Verse DBs have populated these tables.

## Finding: Missing Tariff Version History

The `tariff_version` table defines:
- `version_number`, `effective_start_date`, `effective_end_date`, `status`, `activation_date`

But it is EMPTY. All 37 current tariffs have `valid_from=2026-06-13` — the date the Meter Verse database was initialized. The **historical tariffs used in 2021-2022 are not preserved in any database**.
