## Detailed Analysis

### 1. Tariff Calculation — MERGE into features

**sim_system**: `TariffPlan` (schema.prisma:451–467)
```
model TariffPlan {
  id, projectId, meterType, ratePerUnit, currency, effectiveFrom, effectiveTo, status
}
```
- Single flat rate per unit
- Status: `draft` / `active` / `retired`
- Used by: `TariffService.getEffectiveTariff()` (tariffs/tariff.service.ts:10)

**features**: `Tariff` + `TariffCharge` + `TariffChargeDetail` (schema.prisma:1215–1290)
```
model Tariff { id, tariffCode, tariffName, utilityType, effectiveFrom, effectiveTo }
model TariffCharge { id, tariffId, chargeCode, chargeMode, rateAmount, minCharge, maxCharge, settlementType }
model TariffChargeDetail { id, chargeId, stepFrom, stepTo, stepRate, stepAmount, isPercentage }
```
- All 5 charge modes: STEPS, FLAT, STATIC, PER_UNIT, ZERO
- Min/max charge support
- Tiered steps
- Used by: `TariffEngineService.calculateCharges()` (tariff-engine.service.ts:19) — **prefers features, falls back to sim_system**

**Evidence** (`tariff-engine.service.ts:20–28`):
```typescript
const tariff = await this.prisma.tariff.findFirst({
  where: { utilityType: ..., isActive: true, effectiveFrom: { lte: effectiveDate } }
});
if (!tariff) {
  const flatTariff = await this.prisma.tariffPlan.findFirst({ ... });  // fallback
}
```

**Action**: Remove sim_system TariffPlan. Build CRUD for features Tariff/TariffCharge/TariffChargeDetail. Update TariffService to use features.

### 2. Invoice Generation — MERGE into features

**sim_system**: `Invoice` + `InvoiceLine` + `InvoiceAdjustment` (schema.prisma:488–543)
- Full invoice lifecycle: draft → pending_approval → issued → partially_paid → paid → overdue → cancelled
- Used by all billing controllers

**features**: `InvoiceGenerationBatch` (schema.prisma:1930–1950)
```
model InvoiceGenerationBatch {
  id, batchCode, cycleId, areaId, totalInvoices, successfulCnt, failedCnt, totalAmount, status
}
```
- Tracks batch runs
- Zero code references it

**Action**: Create migration path where `InvoiceGenerationBatch` tracks batch runs from `BillingController.generateInvoices()`. Keep sim_system Invoice as the primary invoice table for now; migrate to `features` schema only in a future phase.

### 3. Customer Ledger — KEEP in sim_system

**sim_system**: `CustomerLedgerEntry` (schema.prisma:576–590)
- Append-only ledger with running balance
- Used by: `LedgerService.addEntry()` (ledger.service.ts:10)
- No duplicate in features

**Action**: No duplication. Keep as-is.

### 4. Payment vs Wallet — KEEP both

**sim_system**: `Payment` + `PaymentAllocation` (schema.prisma:547–574)
- Cash/bank/cheque/card payment tracking
- Oldest-due-first allocation algorithm implemented in `BillingController.createPayment()` (billing.controller.ts:316–351)

**features**: `WalletAccount` + `WalletTransaction` + `WalletBalance` + `WalletAllocation` + `WalletTransfer` (schema.prisma:1415–1522)
- Solar credit wallet (DEPOSIT/WITHDRAWAL/TRANSFER/ALLOCATION/REFUND)
- Designed for solar buyback/sell scenarios
- Solar controller at `solar/solar.controller.ts` uses separate `AreaSolarWalletTransaction`

**Action**: These serve different purposes. Cash payments ≠ solar credits. Keep both.

---

### 5. Settlement — REPLACE with features

**sim_system (core schema)**: `CoreSettlement` (schema.prisma:1119–1136)
```
model CoreSettlement {
  areaId, periodStart, periodEnd, status, totalInvoiced, totalCollected, variance
}
```
- Simple aggregate-level settlement

**features**: 5 settlement tables (schema.prisma:1639–1744)
- `SettlementConfig` — configs with rule types
- `SettlementPeriod` — period definitions
- `SettlementRule` — FIXED_PERCENTAGE/TIERED/FORMULA/MANUAL rules
- `SettlementTransaction` — individual transactions
- `SettlementAllocation` — allocation to targets

**Current controller** (`settlement.controller.ts:15–36`):
```typescript
// WRONG: writes to sim_system.invoice with utilityType='settlement'
return this.prisma.invoice.create({ data: { ..., utilityType: 'settlement', ... } });
```

**Action**: Rewrite settlement controller to write to features settlement tables. Remove sim_system invoice-based settlement hack.

### 6. Billing Cycle — MERGE into features

**sim_system**: `BillingPeriod` (schema.prisma:469–484)
```
model BillingPeriod { projectId, periodCode, startDate, endDate, status: open/in_review/closed }
```
- Used by: `PeriodService` (period.service.ts), `BillingController` (billing.controller.ts:433–457)

**features**: `BillingCycle` + 3 related tables (schema.prisma:1748–1818)
- Approval workflow
- Per-project tracking
- Audit trail
- Status: OPEN/LOCKED/APPROVED/CLOSED/CANCELLED

**Action**: Add utility type + month/year to features BillingCycle. Rewrite PeriodService to use features BillingCycle. Implement state machine.
