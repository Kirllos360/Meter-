Meter Verse Extended Governance Rules

Enforced starting from T022 closeout.
Last updated: 2026-05-29


Rule 1 — Multi-Tool Validation Chain

After every completed task, execute validation in this order:

1. OpenCode validation
2. SpecKit validation
3. OpenSpec validation
4. Open-Interpreter validation
5. Graphify validation

Generate a consolidated validation report.

If any tool identifies: defect, architecture conflict, dependency issue, regression risk, or documentation gap — STOP. Create issue report. Apply fixes. Restart validation sequence.

Repeat until: Acceptance criteria PASS, Build PASS, Tests PASS, Documentation PASS, Architecture PASS.

Target confidence: 95%+ (do not claim 100% certainty).


Rule 2 — Cross-Task Compatibility Validation

After task validation completes, re-test the current task plus all dependency tasks plus all previously completed tasks that directly interact.

Example: T021 must test T020 and any components consuming API Client, React Query, Feature Flags.

Generate: Compatibility Report, Regression Report, Dependency Health Report.


Rule 3 — Phase-Wide Validation

At the end of every task, run a phase-level review covering T001 → Current Task.

Verify: No schema conflicts, no API conflicts, no architectural conflicts, no dependency conflicts, no documentation drift.

Generate: Phase Health Report.


Rule 4 — Documentation Synchronization

After every task, update: README, Architecture documents, Validation reports, Graph memory, Task history, Implementation notes, SpecKit documentation, Deployment documentation.

Verify documentation matches implementation.


Rule 5 — Repository Structure Review

After every task, review repository structure and create/update ROUTE_OF_DATA.md.

Document: Folder structure, Data flow, API flow, Dependency flow, Database flow, Frontend flow.

Recommend improvements if needed. Do not perform large-scale restructuring without approval.


Rule 6 — Backup and Recovery Package

After every task, create backup-files/task-TXXX_TIMESTAMP/ with:

  File   Purpose  
 --- --- 
  checkpoint.md   Session restore point  
  validation-report.md   Multi-tool validation results  
  graph-memory.md   Knowledge graph state  
  task-summary.md   What was done in this task  
  changed-files.txt   List of all files changed/created  
  git-status.txt   git status + git diff output  
  rollback-notes.md   How to revert if needed  
  conversation-memory.md   Full session context  
  resume-context.md   Self-contained AI recovery file  

Timestamp format: YYYYMMDD_HHMMSS


Rule 7 — AI Recovery Context

Maintain resume-context.md — a self-contained file allowing any AI system to continue work.

Include: Project Overview, Current Phase, Completed Tasks, Pending Tasks, Active Branch, Last Commit, Latest Checkpoint, Latest Validation Status, Open Issues, Next Recommended Step.

The file must be self-contained (no external references required).


Rule 8 — Tool and Dependency Inventory

Maintain requirements-inventory.md tracking: Tools, Services, MCP Servers, GitHub Dependencies, NPM Packages, Docker Services, External Systems.

For each item: Purpose, Installation Steps, Repository URL, Version, Usage in Meter Verse. Update after every task.


Rule 9 — GitHub Synchronization Review

After every task, review git status and git diff. Generate GitHub Sync Report.

If PR creation is possible, prepare: Commit Summary, PR Summary, Validation Evidence.

If GitHub permissions are unavailable, document required actions. Do not assume PR creation succeeded.


Rule 10 — Task Closeout Question

Before moving to the next task, ask:

"Validation, compatibility review, documentation synchronization, backup generation, and recovery context creation are complete. Is this sufficient for validation before task closeout and proceeding to the next task?"
