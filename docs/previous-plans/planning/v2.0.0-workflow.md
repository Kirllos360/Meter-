# Meter Verse v2.0.0 — Workflows & State Machines

This document defines 9 core business workflows as Mermaid state diagrams and flowcharts. Each diagram represents a complete, production-grade state machine or process flow.

---

## 1. Meter Lifecycle State Machine (8 Statuses)

The meter lifecycle tracks every meter from inventory through installation, operation, and eventual retirement. Each status transition is audited and may trigger downstream actions (invoice holds, notifications, etc.).

```mermaid
stateDiagram-v2
    [*] --> Available: Meter registered in inventory
    Available --> Assigned: Assigned to customer/connection
    Assigned --> Active: Installation complete & verified
    Active --> Offline: Communication lost (30d no reading)
    Active --> Faulty: Malfunction reported or detected
    Offline --> Active: Communication restored
    Faulty --> Active: Repair completed & tested
    Faulty --> Replaced: Physical replacement ordered
    Replaced --> Active: New meter installed & active
    Active --> Terminated: Contract ended / service closed
    Offline --> Terminated: Contract ended (offline)
    Terminated --> Retired: Meter physically removed
    Retired --> [*]: Scrapped or returned to stock

    note right of Active
        Normal operating state
        Sends readings per schedule
    end note

    note right of Faulty
        Triggers trouble ticket
        Invoice hold applied
    end note

    note right of Terminated
        Final reading taken
        Final invoice generated
    end note
```

### Status Definitions

| Status | Description | Duration |
|--------|-------------|----------|
| `Available` | In inventory, not yet assigned | Indefinite |
| `Assigned` | Allocated to a customer/connection | 1-14 days |
| `Active` | Installed and reporting readings | Months-years |
| `Offline` | No communication for 30+ days | Variable |
| `Faulty` | Malfunction reported, repair pending | 1-30 days |
| `Replaced` | Old meter removed, new one installed | 1-7 days |
| `Terminated` | Service ended, awaiting removal | 1-90 days |
| `Retired` | Physically removed, end of life | Final |

### Transition Table

| From | To | Trigger | System Action |
|------|----|---------|---------------|
| Available | Assigned | Contract activation | Create assignment record |
| Assigned | Active | Installation confirmation | Start reading schedule |
| Active | Offline | 30d no reading received | Set flag, notify team |
| Active | Faulty | Customer report / diagnostic | Create trouble ticket, hold invoices |
| Offline | Active | Reading received | Clear flag, resume billing |
| Faulty | Active | Repair verification | Close ticket, release invoice hold |
| Faulty | Replaced | Replacement order | Create work order |
| Replaced | Active | New meter installed | Link new meter, close old |
| Active | Terminated | Contract closure | Generate final reading/invoice |
| Offline | Terminated | Service closure | Generate final invoice (estimated) |
| Terminated | Retired | Physical removal | Update inventory, archive records |

## 2. Invoice Lifecycle State Machine

```mermaid
stateDiagram-v2
    [*] --> Draft: Generated by billing engine
    Draft --> Issued: Approved & sent to customer
    Issued --> Partial: Partial payment received
    Issued --> Paid: Full payment received
    Issued --> Overdue: Past due date without full payment
    Partial --> Paid: Remaining balance paid
    Partial --> Overdue: Past due on remaining balance
    Overdue --> Paid: Full payment (incl. penalties)
    Overdue --> Overdue: Dunning level escalation
    Issued --> Cancelled: Manual cancellation / credit note
    Draft --> Cancelled: Billing error detected
    Cancelled --> [*]: Closed
    Paid --> [*]: Closed

    note right of Draft
        Not yet visible to customer
        Editable by finance/operator
    end note

    note right of Overdue
        Dunning levels: 7d, 15d, 30d, 60d, 90d
        Each level triggers escalation actions
    end note

    note right of Cancelled
        Requires credit note
        Reverses all balance effects
    end note
```

### Invoice States

| State | Description | Customer Visible |
|-------|-------------|------------------|
| `Draft` | Generated, pending review | No |
| `Issued` | Approved and delivered | Yes |
| `Partial` | Partially paid | Yes |
| `Paid` | Fully paid, settled | Yes |
| `Overdue` | Past due date | Yes |
| `Cancelled` | Voided via credit note | No (hidden) |

## 3. Payment Flow

```mermaid
flowchart TD
    START([Start Payment]) --> SEARCH[Search Customer]
    SEARCH --> SELECT[Select Customer Account]
    SELECT --> VIEW_BAL[View Outstanding Balance]
    VIEW_BAL --> ALLOCATE{Allocate Payment}
    
    ALLOCATE --> FULL[Full Payment<br/>Pay entire balance]
    ALLOCATE --> PARTIAL[Partial Payment<br/>Pay specific amount]
    ALLOCATE --> EXACT[Exact Invoice<br/>Pay specific invoice]
    
    FULL --> CONFIRM{Confirm Amount}
    PARTIAL --> CONFIRM
    EXACT --> CONFIRM
    
    CONFIRM -->|Confirmed| METHOD[Select Payment Method]
    CONFIRM -->|Cancel| START
    
    METHOD --> CASH[Cash]
    METHOD --> CHEQUE[Cheque]
    METHOD --> BANK[Bank Transfer]
    METHOD --> CARD[Card Terminal]
    
    CASH --> RECEIPT[Generate Receipt]
    CHEQUE --> VERIFY{Verify Cheque}
    VERIFY -->|Valid| RECEIPT
    VERIFY -->|Invalid| REJECT[Reject Payment]
    REJECT --> START
    
    BANK --> RECEIPT
    CARD --> RECEIPT
    
    RECEIPT --> PRINT[Print/Email Receipt]
    PRINT --> UPDATE[Update Balance]
    UPDATE --> NOTIFY[Send Notification]
    NOTIFY --> END([Payment Complete])

    subgraph "Validation"
        AMOUNT{Amount > 0?}
        CUSTOMER{Customer Active?}
        AMOUNT -->|No| ERROR1[Error: Invalid Amount]
        CUSTOMER -->|No| ERROR2[Error: Inactive Customer]
    end

    SEARCH --> AMOUNT
    AMOUNT --> CUSTOMER
    CUSTOMER --> SELECT
```

### Payment Allocation Rules

| Allocation Type | Behavior |
|----------------|----------|
| Full Payment | Pays entire outstanding balance across all invoices (oldest first) |
| Partial Payment | Pays a specified amount applied to oldest invoices first |
| Exact Invoice | Pays one specific invoice in full or partial |
| Prepayment | Credits balance before invoice generation |

## 4. Reading Flow

```mermaid
flowchart TD
    START([Capture Reading]) --> METHOD{Reading Source}
    
    METHOD --> MANUAL[Manual Entry<br/>Field operator inputs]
    METHOD --> AMR[AMR Import<br/>Automated meter reading]
    METHOD --> ESTIMATE[Estimate<br/>System-generated]

    subgraph "Capture Phase"
        MANUAL --> VALIDATE{Validate Entry}
        AMR --> VALIDATE
        ESTIMATE --> VALIDATE
        VALIDATE -->|Range Check| RANGE{Within Threshold?}
        RANGE -->|Yes| TREND{Consistent with Trend?}
        RANGE -->|No| FLAG[Flag as Anomaly]
        TREND -->|Yes| ACCEPT[Accept Reading]
        TREND -->|No| FLAG
        FLAG --> REVIEW
    end

    subgraph "Review Phase"
        ACCEPT --> APPROVAL{Approval Required?}
        REVIEW[Review Queue] --> APPROVAL
        APPROVAL -->|Auto-approve| APPROVED[Approved]
        APPROVAL -->|Manual review| SUPERVISOR{Supervisor Decision}
        SUPERVISOR -->|Approve| APPROVED
        SUPERVISOR -->|Reject| REJECTED[Rejected]
    end

    subgraph "Post-Processing"
        APPROVED --> CALC[Trigger Consumption Calculation]
        APPROVED --> UPDATE_METER[Update Meter Status]
        APPROVED --> AUDIT[Log to Audit Trail]
        REJECTED --> REENTRY[Flag for Re-capture]
        REENTRY --> START
    end

    CALC --> END([Reading Complete])
    UPDATE_METER --> END
    AUDIT --> END

    style APPROVED fill:#27ae60,color:#fff
    style REJECTED fill:#e74c3c,color:#fff
    style FLAG fill:#f39c12,color:#fff
```

### Reading Validation Rules

| Rule | Description | Action on Failure |
|------|-------------|-------------------|
| Range Check | Reading must be between 0 and meter max capacity | Flag as anomaly, require review |
| Trend Check | Reading must not deviate > 50% from 3-month rolling average | Flag as anomaly, require review |
| Minimum Check | Reading must be >= last reading (non-decreasing) | Flag for zero/negative consumption review |
| Duplicate Check | No two readings within 24h for same meter | Reject duplicate |
| Bounds Check | Commercial meters max 10,000; residential max 1,000 | Flag if exceeded |

## 5. Water Balance Computation

The water balance compares the main (parent) meter's consumption against the sum of all sub-meters (children) to calculate distribution variance.

```mermaid
flowchart TD
    START([Water Balance Calculation]) --> PERIOD[Select Billing Period]
    PERIOD --> MAIN[Get Main Meter Reading]
    MAIN --> SUBS[Get All Sub-Meter Readings]

    SUBS --> SUM[Calculate Total Sub-Meter Consumption]
    SUM --> GROSS[Calculate Main Meter Consumption]
    
    GROSS --> VARIANCE[Variance = Main - Sum(Subs)]
    VARIANCE --> CHECK{Check Variance}
    
    CHECK -->|Variance < 1%| NORMAL[✅ Normal Balance]
    CHECK -->|Variance 1-5%| ACCEPTABLE[⚠ Acceptable Loss]
    CHECK -->|Variance 5-10%| INVESTIGATE[🔍 Flag for Investigation]
    CHECK -->|Variance > 10%| ALERT[🚨 Critical Alert]

    NORMAL --> REPORT[Generate Balance Report]
    ACCEPTABLE --> REPORT
    INVESTIGATE --> CREATE_TICKET[Create Trouble Ticket]
    ALERT --> CREATE_TICKET
    CREATE_TICKET --> REPORT
    
    REPORT --> DISTRIBUTION{Distribution Type}
    DISTRIBUTION --> PROPORTIONAL[Proportional: Distribute variance by consumption ratio]
    DISTRIBUTION --> SEQUENTIAL[Sequential: Assign to last sub-meter]
    DISTRIBUTION --> MANUAL[Manual: Supervisor adjusts allocation]
    
    PROPORTIONAL --> FINALIZE[Finalize Balance]
    SEQUENTIAL --> FINALIZE
    MANUAL --> FINALIZE
    
    FINALIZE --> END([Balance Complete])

    subgraph "Variance Components"
        LOSS[Physical Loss: Leaks, theft]
        ERROR[Meter Error: Calibration drift]
        TIMING[Timing: Different reading dates]
    end
```

### Variance Thresholds

| Threshold | Level | Action |
|-----------|-------|--------|
| < 1% | Normal | Auto-approve, no action |
| 1-5% | Acceptable | Note in report, monitor trend |
| 5-10% | Investigation Required | Create trouble ticket, assign to inspector |
| > 10% | Critical | Immediate supervisor alert, possible shut-off |

## 6. Consumption Calculation Pipeline

```mermaid
flowchart LR
    subgraph "Input"
        A[Raw Reading<br/>Current: 1250<br/>Previous: 1100]
        B[Meter Config<br/>Multiplier: 1.0<br/>UOM: m³]
    end
    
    subgraph "Step 1: Consumption"
        C[Consumption = Current - Previous]
        C --> D{Threshold Check}
        D -->|Normal| E[Consumption: 150 m³]
        D -->|Zero| Z[Zero Consumption]
        D -->|Negative| N[Negative Flag]
        D -->|Spike| S[Spike Flag > 2x avg]
    end
    
    subgraph "Step 2: Tariff Lookup"
        E --> F[Lookup Tariff Schedule]
        F --> G{Tariff Type}
        G -->|Flat| H[Rate: 3.50 EGP/m³]
        G -->|Tiered| I[Tier Check]
        I --> J[Tier 1: 0-50 @ 2.00]
        I --> K[Tier 2: 51-100 @ 3.50]
        I --> L[Tier 3: 101+ @ 5.00]
        G -->|Seasonal| M[Season: Summer<br/>Rate: 4.00 EGP/m³]
    end
    
    subgraph "Step 3: Charge Calculation"
        H --> N[Line Item: Consumption]
        J --> N
        K --> N
        L --> N
        M --> N
        N --> O[Amount: 150 × 3.50 = 525.00 EGP]
    end
    
    subgraph "Step 4: Additional Charges"
        O --> P[Service Fee: 10.00]
        O --> Q[Wastewater: 25% × 525 = 131.25]
        O --> R[Late Penalty: if applicable]
    end
    
    subgraph "Step 5: Invoice Line Items"
        P --> S[Line 1: Consumption - 525.00]
        Q --> S
        R --> S
        S --> T[Line 2: Service Fee - 10.00]
        T --> U[Line 3: Wastewater - 131.25]
        U --> V[Total: 666.25 EGP]
    end
    
    V --> END([Invoice Line Items Generated])
```

### Calculation Formula

```
Consumption = (Current Reading - Previous Reading) × Meter Multiplier

If Consumption < 0:     Flag as negative, require manual review
If Consumption = 0:     Flag as zero consumption, check for fault
If Consumption > 2×Avg: Flag as spike, check for leak or error

For each consumption block:
  Block Amount = Block Volume × Block Rate

Total Charge = Σ(Block Amounts) + Service Fee + Wastewater Fee + Penalties
Wastewater Fee = Consumption Charge × Wastewater Rate (25%)
```

## 7. Trouble Ticket Flow

```mermaid
stateDiagram-v2
    [*] --> New: Ticket created by customer or system
    New --> Assigned: Dispatcher assigns to technician/team
    Assigned --> InProgress: Technician starts work
    InProgress --> OnHold: Waiting for parts / info
    OnHold --> InProgress: Parts received / info provided
    InProgress --> Resolved: Work completed, solution applied
    Resolved --> Closed: Customer confirms satisfaction
    Resolved --> Reopened: Customer reports issue persists
    Reopened --> Assigned: Re-assigned for follow-up
    Closed --> [*]: Ticket archived
    New --> Cancelled: Duplicate / invalid report
    Assigned --> Cancelled: Cannot be resolved

    note right of New
        Priority assigned: S0/S1/S2/S3
        SLA clock starts
        Notification sent to dispatcher
    end note

    note right of InProgress
        Technician updates status
        Logs work performed
        Attaches photos if applicable
    end note

    note right of Resolved
        Customer notified
        Satisfaction survey sent
        7-day warranty period starts
    end note
```

### Ticket Priority Matrix

| Priority | Response SLA | Resolution SLA | Example |
|----------|-------------|----------------|---------|
| S0 - Critical | 15 min | 4 hours | No water, major leak, fire |
| S1 - High | 1 hour | 24 hours | Meter burst, no billing |
| S2 - Medium | 4 hours | 72 hours | Faulty meter, billing error |
| S3 - Low | 24 hours | 7 days | General inquiry, cosmetic issue |

    FINALIZE --> ISSUE[Step 4: Issuance Phase]
    
    subgraph "Issuance Phase"

        DELIVER --> EMAIL[Email PDF]
        DELIVER --> SMS[SMS with Link]
        DELIVER --> PRINT[Print & Mail]

    SENT --> DUE[Step 5: Due Date Phase]

    OVERDUE --> COLLECT[Step 6: Collection Phase]
    
    subgraph "Collection Phase"
        COLLECT --> DUNNING{Dunning Level}
        DUNNING -->|Level 1: 7d| REMINDER1[Reminder Notification]
        DUNNING -->|Level 2: 15d| REMINDER2[Second Notice + Fee]
        DUNNING -->|Level 3: 30d| REMINDER3[Final Notice + Penalty]
        DUNNING -->|Level 4: 60d| DISCONNECT[Disconnection Warning]

    REMINDER1 --> PAY{Payment Received?}
    REMINDER2 --> PAY
    REMINDER3 --> PAY
    DISCONNECT --> PAY
    ESCALATE --> PAY
    
    PAY -->|Yes| CLOSE[Close Bill Cycle]
    PAY -->|No| CONTINUE[Continue Dunning]
    CONTINUE --> DUNNING
    
    CLOSE --> REPORT[Generate Cycle Report]
    REPORT --> END([Bill Cycle Complete])

## 9. Availability Plans — Switching Logic

```mermaid
stateDiagram-v2
    [*] --> Monitoring: System Running

    state Monitoring {
        [*] --> CheckHealth
        CheckHealth --> Healthy: All DBs reachable
        CheckHealth --> Degraded: Load > 70% or latency > 500ms
        CheckHealth --> Critical: Area DB unreachable
    }

    state "Full Plan" as FULL {
        [*] --> AllActive
        AllActive --> NormalOps
        NormalOps --> SyncWrites
        SyncWrites --> AllActive
    }

    state "Safety Plan" as SAFETY {
        [*] --> DegradedReads
        DegradedReads --> CacheMode
        CacheMode --> AsyncQueue
        AsyncQueue --> PeriodicSync
        PeriodicSync --> HealthCheck
        HealthCheck --> DegradedReads
        HealthCheck --> [*]: Conditions clear
    }

    state "Failover Plan" as FAILOVER {
        [*] --> PromoteReplica
        PromoteReplica --> StandbyActive
        StandbyActive --> QueueWrites
        QueueWrites --> DrainQueue
        DrainQueue --> SyncToPrimary
        SyncToPrimary --> VerifyReplica
        VerifyReplica --> StandbyActive
        VerifyReplica --> [*]: Primary restored
    }

    Healthy --> FULL: Normal operation
    FULL --> Monitoring: Periodic health check

    Degraded --> SAFETY: Auto-switch at 70% load
    SAFETY --> Monitoring: Load drops below 50%
    SAFETY --> FULL: Manual override

    Critical --> FAILOVER: Auto-switch on DB failure
    FAILOVER --> Monitoring: Primary restored & synced
    FAILOVER --> SAFETY: Manual degrade

    note right of FULL
        10 TCP channels active
        100 HTTP routes
        All 17 DBs read-write
        Synchronous writes
    end note

    note right of SAFETY
        8 TCP channels active
        70 HTTP routes (reports/admin limited)
        Area DBs read-only
        Async write queue
        Cache hit mode
    end note

    note right of FAILOVER
        6 TCP channels active
        40 HTTP routes (read-only)
        Replica promoted to primary
        Queue missed writes
        RTO: 120s / RPO: 60s
    end note
```

### Switching Logic (Pseudocode)

```
monitor():
  every 5s:
    for each area_db:
      latency = ping(area_db)
      load = get_cpu_load()
      
      if latency > 1000ms or not reachable:
        trigger(FAILOVER, area_db)
      elif load > 70% or latency > 500ms:
        trigger(SAFETY)
      else:
        if current_plan != FULL and conditions_clear_for(300s):
          trigger(FULL)

trigger(FULL):
  enable_all_tcp_channels()
  enable_all_http_routes()
  set_all_dbs_read_write()
  flush_async_queue()
  log("Switched to Full Plan")

trigger(SAFETY):
  disable_tcp_channels([07, 08, 09, 10])
  disable_http_routes([/reports/*, /admin/*])
  set_area_dbs_read_only()
  enable_async_write_queue()
  enable_cache_hit_mode()
  log("Switched to Safety Plan")

trigger(FAILOVER, area):
  promote_replica(area)
  update_routing_table(area, replica_endpoint)
  disable_tcp_channels([04, 05, 06, 07, 08, 09])
  disable_http_routes([/payments/*, /invoices/*, /readings/*])
  start_write_queue(area)
  notify_ops_team(area)
  log("Switched to Failover Plan for", area)
```
